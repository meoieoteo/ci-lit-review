# Literature Review Synthesis: Lessons for the Problem Statement

The main synthesis is that the literature supports the core intuition, but the hard part is not model architecture. It is target definition, label validity, timing, validation, and care-pathway bias.

## 1. Treat the Model Target as the Central Design Decision

The Lu, John, Zhou, and Grammenos papers all point to the same problem: "dementia prediction" can mean several different things.

For our problem statement, we should explicitly distinguish:

- Current unmet cognitive care need
- Currently present but undocumented cognitive impairment
- Future cognitive impairment
- Future coded dementia diagnosis
- Future referral, testing, or care pathway

Those are not interchangeable. Zhou is especially useful here: models trained on VES-13 versus eFI labels learned different frailty constructs. The dementia analogue is diagnosis code versus cognitive screen versus neurology referral versus chart-reviewed impairment.

Extension idea: create a `label_definition_reviewer.md` specialist agent focused entirely on label source, timing, leakage, and downstream-action dependence.

## 2. Use EHR Text as a Signal Source, But Prefer Interpretable Intermediate Concepts Where Possible

Adejumo, Khan, Osman, Huang, Alsentzer, GatorTron, and ClinicalBERT all support the idea that free text contains clinically useful information missing from structured data.

For our problem, the strongest extension is not just "throw all notes into an LLM." A better first strategy is:

- Extract cognitive, functional, behavioral, caregiver, adherence, frailty, falls, delirium, social, and appointment-navigation concepts from notes.
- Preserve note date and note type.
- Model these concepts longitudinally before the cardiology encounter.
- Compare concept features against raw note embeddings.

Khan's eFI approach is especially relevant: use NLP to create clinically interpretable deficit features, then use those features in downstream risk models.

Extension idea: build a cognitive/frailty note-concept inventory before picking the final model.

## 3. The Index Date Must Be the Cardiology Encounter Start

The problem statement already says this, and the literature makes it more important. Many prediction papers blur index date, observation window, and outcome window. John shows that published dementia models often fail to report these fields well enough for external validation.

For our study design, every row should be anchored to:

```text
patient_id + cardiology_encounter_id + encounter_start_time
```

Allowed inputs:

```text
all EHR data strictly before encounter_start_time
```

Disallowed inputs:

```text
anything during or after the cardiology encounter
```

Extension idea: make the summarizer/reviewer schema track whether a paper clearly separates `index_date`, `observation_window`, and `prediction_horizon`.

## 4. Downstream-Action Bias Is Probably the Core Methodological Contribution

The problem statement already identifies the issue: two equivalent patients may diverge because one gets referred and one does not. Lu, John, and the target-trial/cohort-design papers reinforce that labels derived from routine care are partly labels of the care pathway.

This matters most if the label is future diagnosis, future cognitive testing, or specialist referral. Those outcomes are not pure disease states.

Extension idea: define a dataset-construction strategy that separately models:

- Cognitive outcome evidence
- Opportunity for diagnosis
- Downstream action after cardiology
- Follow-up completeness
- Competing censoring or death
- PCP, neurology, geriatrics, psychiatry, or memory clinic contact

This could become one of the project's real contributions.

## 5. External Validation and Calibration Should Be Designed in From Day One

Lu found high risk of bias across EHR dementia prediction models. John found many models cannot even be externally validated from published details. These are warnings.

For our model, AUROC alone is insufficient. We should plan for:

- Temporal validation
- Site or practice validation if possible
- Calibration plots, intercepts, and slopes
- Threshold-specific sensitivity, specificity, PPV, and NPV
- Decision-curve or net-benefit analysis tied to cardiology workflow

Extension idea: build a `validation_reviewer.md` agent derived from Lu and John, with required fields for transportability, calibration, thresholding, and clinical utility.

## 6. Start Simple Before Using Large Language Models End to End

The model literature gives a useful architecture ladder:

1. Structured baseline model
2. Structured plus note-derived concepts
3. Structured plus note embeddings
4. Longitudinal sequence model
5. LLM-assisted extraction or augmentation
6. End-to-end LLM only if justified

Adejumo and Khan suggest concept extraction can be powerful and auditable. BEHRT, Med-BERT, RETAIN, and ClinicalBERT suggest longitudinal representation learning is plausible. GatorTron and Med-PaLM suggest large models are useful, but their safest role may be extraction, summarization, or weak-label support, not autonomous clinical scoring.

## 7. Sequential Diagnosis Papers Belong Later, But They Help Frame the Future

Aronsson, Janisch, Vivar, Poh, Horvitz, and Kaelbling are not directly about the first model in the problem statement. They are useful for the later "what should the cardiologist do next?" question.

For now, they should inform design boundaries:

- Our first model scores risk before the encounter.
- It should not recommend next tests yet.
- Its output should be compatible with later value-of-information or feature-acquisition work.

## Practical Next Steps

I would extend the workflow with three immediate artifacts:

1. `templates/study_design.md`
   A concrete design skeleton for our own study: index date, observation window, candidate labels, outcome windows, censoring, validation.

2. `agents/specialist_reviewers/label_definition_reviewer.md`
   Focused on whether a paper's label means disease, documentation, diagnosis opportunity, referral, or downstream action.

3. `agents/specialist_reviewers/validation_reviewer.md`
   Focused on validation design and reporting, with fields for external validation, transportability, calibration, thresholding, deployment fit, and reproducibility.

The highest-value literature lesson is this: our differentiator should probably be careful cohort and label construction for pre-cardiology cognitive-risk scoring using longitudinal EHR text, not just applying a newer model.
